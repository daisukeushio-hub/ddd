# Sansan Scraper Debug Report

## 概要

このスクレイパーは、Sansan の企業検索画面で

- 売上レンジ
- 業種の大分類 / 中分類

を条件に検索し、検索結果一覧を CSV に蓄積するためのもの。

当初は以下の問題があった。

- 業種 CSV の読み込み不整合
- 1 ページ目しか取れない
- 検索条件を変えても結果が切り替わらないことがある
- 中分類の選択が不安定
- Selenium / ChromeDriver の接続断で途中停止する
- 主業 / 従業の業種列取得が欠ける
- 出力列が不足している

現時点では、

- 複数ページ取得
- 出力列の大半
- 売上レンジ指定実行
- `resume` 再開
- 重複排除

は実用レベルまで来ている。

未解決の主問題は、

- 条件切替時に別の業種 selector を触る / 読むことがある
- 条件を変えたつもりでも、前条件と同じ結果セットを再読する疑いがある

の 2 点。


## これまでの問題と対応

### 1. `industries.csv` のヘッダー不一致

症状:

- `industries.csv のヘッダーは 大分類,中分類,小分類 が必要です`

原因:

- BOM と現行 CSV 仕様のずれ
- 実際の CSV は `大分類,中分類` ベース

対応:

- `utf-8-sig` 読み込みに変更
- `小分類` なしでも動くように変更

結果:

- 解決


### 2. 検索結果が 1 ページ目しか取れない

症状:

- `page=1` だけ取得して終了
- `831件` ある条件でも 30 件しか取れない

原因:

- ページ送りの通常クリックが Sansan 側の実装と噛み合っていなかった
- `href` 直叩きも期待通りに動かなかった

試したこと:

- Selenium 通常クリック
- JS click
- `mousedown` / `mouseup` / `click` の順発火
- `href` への直接遷移
- `fetch()` で次ページ HTML を取り、結果一覧 DOM を差し替える fallback

実際に機能したもの:

- `fetch()` + DOM 差し替え fallback
- 一部ページではイベント click

結果:

- 複数ページ取得は解決
- 例: `10億-30億|農業、林業>農業` で `831件 / 28ページ` 取得確認済み


### 3. 出力列不足 / 業種列欠落

症状:

- もともとの CSV では欲しい列が足りない
- `会社名` がない
- `主業 / 従業` の業種列が欠ける

対応:

- 列構成をユーザー要件に合わせて変更
- `会社名` 列を先頭に追加
- 次の DOM クラスから主業 / 従業を取得
  - `sansan-industrial-classification-1-division`
  - `sansan-industrial-classification-1-major-group`
  - `sansan-industrial-classification-2-division`
  - `sansan-industrial-classification-2-major-group`

結果:

- 列追加は解決
- 業種列も取得できるようになった
- ただし Sansan 検索自体が `主業 / 従業` を厳密に分けていないため、検索結果そのものは混在し得る


### 4. 売上レンジを分割実行できない

症状:

- 1 回の全実行が約 40 時間規模

対応:

- `--sales-range` オプションを追加
- 13 本の売上レンジを指定実行可能にした

使える値:

- `5000man-1oku`
- `1oku-3oku`
- `3oku-5oku`
- `5oku-10oku`
- `10-30`
- `30-50`
- `50-100`
- `100-300`
- `300-500`
- `500-1000`
- `1000-3000`
- `3000-5000`
- `5000-1cho`

結果:

- 解決


### 5. Selenium の `invalid session id` / `ConnectionResetError`

症状:

- 長時間実行中に Chrome / driver セッション断
- `invalid session id`
- `ConnectionResetError: [WinError 10054]`

対応:

- `state.json` にカーソル保存
- `--resume` 再開
- 一部操作を `retry_call()` 化

結果:

- 根絶はしていない
- ただし再開前提では運用可能


### 6. 中分類選択が不安定

症状:

- 特定の中分類で `failed to select option ...`
- 例:
  - `廃棄物処理`
  - `自動車整備`
  - `機械等修理`
  - `人材サービス`
  - `警備`

原因:

- Sansan の `select2` UI と実 `select` がずれる
- 中分類候補は大分類依存で Ajax 取得
- 一部ケースで候補コードが取れない

試したこと:

- `select` に既存 `option` があれば直接選択
- `select2` を開いて検索して選択
- Enter 確定
- `data-url` を使って Ajax 候補取得
- モーダル HTML 取得から候補コード抽出
- 手動 override マップ

実際に機能したもの:

- ケースによってバラつく
- 現実的には手動 override が一番安定

手動 override で追加済み:

- `廃棄物処理` -> `R01`
- `自動車整備` -> `R02`
- `機械等修理` -> `R03`
- `人材サービス` -> `R04`
- `警備` -> `R05`
- `業務請負、アウトソーシング` -> `R06`
- `物品賃貸` -> `R07`
- `郵便局` -> `R08`
- `協同組合` -> `R09`
- `その他の事業サービス` -> `R98`

結果:

- 一部は解決
- ただし一般解にはまだ至っていない


### 7. 検索結果が前条件から切り替わっていない疑い

症状:

- 例:
  - `50億-100億|サービス>機械等修理`
  - `50億-100億|サービス>人材サービス`
  - `50億-100億|サービス>警備`

で、ページごとの `new_sig` の並びが同じ

例:

- `9531879302186`
- `4625181515133`
- `5897407439033`
- `2479061128127`
- `4403076485406`

が別条件でも繰り返し出る

意味:

- 単なる高重複率の可能性もあるが、
- 条件切替に失敗して前条件の結果セットを再読している疑いが強い

試したこと:

- 検索後に `row_signature` / `pager_label` の変化を待つ
- 前条件の `result marker` と同一なら失敗扱いにする
- 大分類 / 中分類の選択後に、選択状態が意図通りか検証

ここで新たに分かったこと:

- 検証時に `宿泊、飲食サービス` が selected と読まれる
- 意図した `サービス` ではない
- つまりページ内の別 selector を見ている / 触っている

現在の未解決点:

- `対象業種` ブロックの selector 操作対象を、さらに厳密に限定する必要がある
- 「どの select を変更したか」をそのまま返し、その要素だけを検証する設計に寄せる必要がある


## なぜこんなに詰まるのか

理由は単純で、対象が「普通の HTML フォーム」ではないから。

この画面は、

- `select2` ベースの見た目
- 実体 `select`
- Ajax 候補取得
- 大分類に依存する中分類
- 結果一覧の Ajax 差し替え
- ページ送りも部分更新

という複数の状態機械が重なっている。

つまり、こちらが 1 つ直すたびに

- 触るべき要素
- 反映確認の仕方
- 次の非同期更新

が別の層で壊れる。

今回の難しさは、

- DOM が 1 個だけではない
- 同じような selector が複数ある
- 画面上見えている文字と、実際に検索条件へ使われている値が一致しないことがある

という点にある。


## いま実際に機能しているもの

- CSV の読み込み
- 売上レンジ指定
- 複数ページ取得
- CSV 出力
- 重複排除
- `resume`
- 多くの業種での検索と蓄積
- 一部中分類の override による突破


## いま未解決のもの

- `対象業種` の selector を確実に 1 個に限定すること
- 条件切替後に本当に別結果になったことの保証
- 全中分類を手動 override なしで安定選択する一般解


## 次に取るべき方針

現実的には 2 つある。

### 方針A: 実行優先

- 詰まった中分類だけ `option value` を DevTools で拾う
- 手動 override を追加
- 50-100 の本番取得を先に完走させる

向いているケース:

- とにかく今データを集めたい
- 完璧な一般解より進捗優先


### 方針B: 根本修正

- `対象業種` ブロックの実 DOM を DevTools で詳しく確認
- 大分類 / 中分類の selector 構造を固定
- 選択対象要素を 1 個に絞り込む
- その要素の `value` / `option:checked` だけで検証する

向いているケース:

- 今後も繰り返し使う
- 手動 override を増やしたくない

特に:

## 結論

このタスクは「Codex に向いていない」というより、

- Sansan 側 UI の癖が強い
- DevTools での実 DOM 観察が必要
- 実行と検証の往復回数が多い

という種類の難しさがある。

なので、次の一手はこれが最も合理的。

- データ取得を優先するなら手動 override を増やす
- 根本修正を優先するなら DevTools 情報を取って selector を確定する

どちらでも継続可能。
